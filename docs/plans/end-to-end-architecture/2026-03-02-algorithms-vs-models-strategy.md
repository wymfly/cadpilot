# 战略抉择：跳过算法直接使用模型替代的可行性分析

在面临“是否要跳过传统的 C++ / Python 几何算法，直接使用深度学习模型替代底层计算（如网格修复、布尔运算、切片）”这一战略抉择时，我们需要从技术成熟度、容错率以及产品化落地三个维度进行极度冷静的评估。

## 核心结论：现阶段（2026年）绝不能“全盘跨越”

**简短回答：不要完全跳过算法。我们必须采取“以算法保底，以模型攻坚”的混合架构（Hybrid Architecture）。**

## 为什么不能直接“纯模型化”？

### 1. 绝对精度与“差之毫厘”的物理现实
*   **传统算法（如 CadQuery / Manifold3D）**：是基于严格的浮点数学方程。当你要求它开一个直径 `12.00mm` 的孔，它计算出来的边界就是精确的 `12.00mm`。它能保证两块金属在物理世界严丝合缝地装配在一起。
*   **AI 模型（如 DeepSDF / Neural-Pull）**：是基于概率和拟合的隐式场。当你让模型去“推断”一个孔，它生成的可能是 `11.95mm` 或 `12.08mm`。对于 3D 打印的机械装配而言，**0.1mm 的误差就意味着废品**。AI 的“近似正确”在视觉展示上足够完美，但在工程物理上是致命的。

### 2. 黑盒与可解释性（Debugging Hell）
*   **传统算法**：如果布尔运算失败了，抛出的异常是 `Coplanar face intersection error`，你可以通过写代码在两个面之间加一个 `0.001mm` 的偏移量来绕过这个 Bug。
*   **AI 模型**：如果 Neural CSG 把模型融合得像一块融化的口香糖，你得到的是不可解释的。你无法通过改代码来修复它，只能说“这个案例在它的潜空间里没有泛化好”。这就导致了客户报 Bug 时，你根本无从下手修复。

### 3. 计算资源的非对称消耗
*   **传统算法**：比如用 `Shapely` 计算某一层切片的轮廓内缩（Offset），在 CPU 上只需要 `0.01` 秒。
*   **AI 模型**：如果为了避免写轮廓内缩算法，强行引入一个 GNN (图神经网络) 去端到端预测路径，它可能需要在一张昂贵的显卡上推理 `5` 秒钟。这在商业 SaaS 部署上是极其不划算的。

---

## 混合架构策略（Hybrid Strategy）应该怎么玩？

我们不能全盘跨越，但可以在某些**“算法实在搞不定（极易崩溃）”**的特定节点上，引入 AI 模型作为“特效药”。

### 策略 A：用模型做“宏观生坯”，用算法做“微观手术”
这正是我们目前规划的路线：
1.  **AI 模型负责发散性创造**：让 Hunyuan3D 去无中生有，让 LLM 去看图写草稿。
2.  **传统算法负责收敛与定规**：拿到 AI 的生坯后，立刻用 `MeshLib`（体素算法）把它强行固化为水密实体，再用 `Manifold3D` 极其精准地挖掉需要装配的螺纹孔。

### 策略 B：用模型做“算法的参数调优师”
与其让 AI 模型去替代切片软件（自己算怎么走线），不如让大语言模型（LLM）去阅读 3D 模型的几何特征（壁厚、悬垂角度），然后**聪明地生成一份最优化参数的 JSON 文件**，最后调用 C++ 写的 CuraEngine 去执行切片。

### 策略 C：“生死劫”节点的双轨并行（Fallback 机制）
针对我们在前置研究中发现的最难搞的节点——**模型清洗与修复 (`mesh_healer`)**。
由于传统算法（体素化）容易丢失边缘且内存极易爆掉，这里是**唯一推荐深度引入 AI 模型（如 Neural-Pull 或 MeshAnything V2）的地方**。
*   **执行逻辑**：当一个烂网格进来，先让算法库（MeshLib）试着修。如果 MeshLib 报 `OOM` 或者修出来的东西依然非流形，立即切换路由（在 LangGraph 中很容易实现），把这个烂模型丢给云端的 Neural-Pull 模型去重建一个全新的、完美的拓扑外壳。

---

## 总结
跳过算法直接用纯 AI 模型（Pure Neural Pipeline），那是实验室用来发顶会的屠龙技。
对于我们要落地的商业工程（需要面对刁钻的客户和真实的 3D 打印机），**最高明的做法不是“用模型淘汰算法”，而是用 LangGraph 把它们像齿轮一样咬合在一起**。让模型解决“不知道怎么画”的问题，让算法保证“打出来能用”的底线。
