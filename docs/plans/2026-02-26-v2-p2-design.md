## P2.1 — 自动化评测基准

### 目标

建立可重复运行的评测基准，量化 V2 管道在图纸解析准确率、几何匹配率、代码生成质量三个维度的表现，为后续优化提供数据支撑。

### 数据集结构

```
benchmarks/
├── v1/                            # 第一版基准数据集
│   ├── case_001_flange/
│   │   ├── drawing.png            # 工程图纸（输入）
│   │   ├── expected_spec.json     # 手工标注的 DrawingSpec
│   │   └── expected_bbox.json     # 预期包围盒 {xlen, ylen, zlen, tolerance_pct}
│   ├── case_002_shaft/
│   │   └── ...
│   └── manifest.json              # 数据集元信息（case 列表、版本、来源）
└── v2/                            # 第二版（更复杂的零件）
    └── ...
```

`expected_spec.json` 格式（与 `DrawingSpec` 对齐）：

```json
{
  "part_type": "rotational_stepped",
  "overall_dimensions": {"max_diameter": 100, "total_height": 30},
  "base_body": {
    "method": "revolve",
    "profile": [
      {"diameter": 100, "height": 10, "label": "base_flange"}
    ],
    "bore": {"diameter": 10, "through": true}
  },
  "features": [
    {"type": "hole_pattern", "count": 6, "diameter": 10, "pcd": 70}
  ]
}
```

`expected_bbox.json` 格式：

```json
{
  "xlen": 100.0,
  "ylen": 100.0,
  "zlen": 30.0,
  "tolerance_pct": 10
}
```

### 评测指标

| 指标 | 计算方式 | 目标值 |
|------|---------|-------|
| **编译通过率** | 生成代码可执行 STEP / 总 case 数 | ≥ 80% |
| **类型识别准确率** | `spec.part_type == expected.part_type` / 总 case | ≥ 85% |
| **参数提取准确率** | 关键尺寸在 10% 误差内的比例（按 `validate_code_params`）| ≥ 70% |
| **几何匹配率** | `validate_bounding_box` 通过 / 编译成功 case | ≥ 60% |
| **平均耗时（s）** | 各 stage 耗时均值 | 记录基线 |
| **平均 Token 消耗** | 每 case VL+Coder 输入/输出 token 之和 | 记录基线 |

### 运行方式

```bash
# 运行完整评测
python -m cad3dify.benchmark run --dataset benchmarks/v1/ --output results/

# 只评测 spec 解析（无代码执行，快速验证）
python -m cad3dify.benchmark run --dataset benchmarks/v1/ --spec-only

# 指定并行度
python -m cad3dify.benchmark run --dataset benchmarks/v1/ --workers 4
```

### 数据集来源建议

1. **GrabCAD 公开图纸**：下载机械零件图纸（法兰盘、齿轮、支架等），手工标注 expected_spec
2. **教材图纸**：机械制图教材中的练习图（版权清晰）
3. **合成生成**：用 CadQuery 生成已知参数的零件，再渲染为 2D 工程图（可自动标注）

建议初始数据集包含 20 个 case，覆盖所有 `PartType`（每类 3-4 个，难度梯度从简单到中等）。

## P2.2 — 成本优化

### 问题分析

当前 V2 管道每次运行的主要 Token 消耗：

| Stage | 模型 | 估算 Token/次 | 必要性 |
|-------|------|-------------|-------|
| Stage 1: VL 分析 | qwen-vl-max | 2,000 in + 800 out | 始终需要 |
| Stage 4a × N: VL 比较 | qwen-vl-max | 1,500 in + 300 out | 可跳过 |
| Stage 4b × N: Coder 修复 | qwen-coder | 2,500 in + 800 out | 可降级 |

当 `num_refinements=3` 时，最坏情况消耗约 **15,000 token**（主要是 VL 比较）。
P0/P1 的三层防线（静态校验 → 包围盒 → VL）已大幅减少不必要的 VL 调用，P2.2 在此基础上进一步优化。

### 优化策略

#### ~~策略 1：早退出（Early Exit）~~ — 已否决

> **否决原因：** SmartRefiner 已重构为零风险模式（`b074d26`），Layer 1/2 不再
> 作为 gate 决策。VL 始终运行是质量保障的基础。任何跳过 VL 的优化都存在质量风险，
> 不符合"成本优化不能影响质量"的原则。
>
> 当前 VL PASS 时 SmartRefiner 返回 `None`，pipeline 层 `break` 退出循环——
> 这是唯一允许的提前退出方式（VL 亲自判定 PASS，非静态检查代替）。

#### 策略 2：模型降级（Model Downgrade）

```
Round 1:  qwen-vl-max（全能力，找大问题）
Round 2+: qwen-vl-plus（降级，处理剩余细节）
```

Coder 修复：
```
简单修复（只有数值参数变更）→ qwen-coder-plus（更快更便宜）
复杂修复（结构性变更）→ qwen-coder（全能力）
```

判断"简单修复"的规则：fix_instructions 中无"缺少"、"添加"、"结构"等关键词，只有数值差异。

#### 策略 3：Token 用量监控

在 `pipeline.py` 的每个 LLM 调用处注入 token 追踪：

```python
# 每次 invoke 后记录
pipeline_stats = {
    "stage_1_tokens": {"in": 2100, "out": 820},
    "stage_4_rounds": [
        {"round": 1, "compare_tokens": {"in": 1480, "out": 290}, "fix_tokens": None},
        {"round": 2, "compare_tokens": None, "status": "PASS"},
    ],
    "total_tokens": 4890,
    "total_duration_s": 42.1,
}
```

输出到 `pipeline_stats.json`，供评测基准和成本分析使用。

#### 策略 4：缓存（针对相同输入）

对同一图片的 Stage 1 分析结果进行内容哈希缓存（TTL 1 小时）：

```python
cache_key = hashlib.sha256(image_data.data.encode()).hexdigest()[:16]
cached = _load_cache(cache_key)
if cached:
    spec, reasoning = cached["spec"], cached["reasoning"]
else:
    # 正常调用
    spec, reasoning = analyze(image_data)
    _save_cache(cache_key, spec, reasoning)
```

适用场景：开发调试时同一图纸反复测试（节省 Stage 1 的 VL 消耗）。

### 实施优先级

| 策略 | 实施难度 | 预期节省 | 优先级 |
|------|---------|---------|-------|
| ~~策略 1：早退出~~ | ~~低~~ | ~~40-60% VL~~ | **已否决**（质量风险） |
| 策略 3：Token 监控 | 低（LangChain callback） | 无节省，但提供数据 | P0 |
| 策略 2：模型降级 | 中（需参数化 model_name） | 20-30% 成本 | P1 |
| 策略 4：缓存 | 中（需文件系统/Redis） | 调试场景 90% | P2 |

### 实施依赖

- Token 监控需要 LangChain 的 `BaseCallbackHandler` 机制
- 模型降级需要将 `ChatModelParameters.from_model_name()` 改为可在 pipeline 层配置
- 早退出可直接在当前 pipeline.py 中实现，无外部依赖

## 总结

| 方向 | 核心价值 | 实施工作量 |
|------|---------|----------|
| P2.1 评测基准 | 量化当前管道质量，为优化提供数据 | ~3 人天（含数据集构建） |
| P2.2 成本优化 | 减少 API 调用成本，提升响应速度 | ~1-2 人天（策略 1+3 优先） |

建议实施顺序：P2.2 策略 3（零风险数据采集）→ P2.1 数据集构建（需人工标注）→ P2.2 策略 2（数据支撑后再降级）。
策略 1 已否决。质量提升方向见 `2026-02-26-v2-quality-improvement-directions.md`。
