# cad3dify V3：自然语言到工业级 3D 打印文件 — 需求提案

> **版本**: v1
> **日期**: 2026-02-26
> **状态**: 待审批

---

## 动机

### 核心矛盾

cad3dify V2 已实现"工程图纸 → STEP 文件"的基本管道，但存在三个根本性限制：

1. **输入形态单一**：仅支持 2D 工程图纸，无法接受自然语言、参数表、参考图片等更直观的输入方式。普通用户不会绘制工程图纸。

2. **精度上限受 LLM 约束**：V2 精度 ≈ VL 准确率(90%) × LLM 代码正确率(70%) ≈ 63%。LLM 同时承担"理解意图"和"精确数值计算"两个本不擅长结合的任务，精度天花板难以突破。

3. **无 UI 交互**：全部流程在后端管道中运行，用户无法参与参数确认、预览调整、知识库管理等关键环节。缺乏"人在环路"使得精度和用户体验都受限。

### V2 当前能力基线

| 能力 | V2 现状 | 差距 |
|------|---------|------|
| 输入 | 仅 2D 工程图纸 | 不支持自然语言/参数表/参考图片 |
| 输出 | 仅 STEP | 不支持 STL/3MF（无法直接打印） |
| 精度 | 5-10% 误差（P0/P1 优化后 3-5%） | 工业级要求 < 0.5% |
| 验证 | 参数校验 + 包围盒 + VL 比较 | 缺少体积/拓扑/可打印性检查 |
| 交互 | 无 UI | 无参数确认/预览/知识库管理 |
| 代码生成 | LLM 自由生成 | 无参数化模板兜底 |

### 外部生态评估结论

深度评估 8 个开源项目（CADAM、CAD-Coder、Text-to-CadQuery、Text2CAD、CAD-MLLM、PartCAD、Build123d）后确认：

- **无可替代项目**：没有一个项目同时覆盖"生成 + 验证 + 修复"
- **CadQuery/OCCT 是正确选择**：唯一兼具工业级精度（BREP/STEP/NURBS）和 LLM 训练生态（~450K 标注数据）的 CAD 框架
- **外部数据/方法可整合**：Text-to-CadQuery 170K 数据集（RAG）、NeurIPS 2025 的 SFT+GRPO 训练方法、CADAM 的参数滑块 UI 设计思路

详见 `docs/plans/2026-02-26-v3-tech-research.md`。

## 目标架构

### 终极目标

**通过自然语言（可附加参数描述、参考图像），生成可直接打印的工业级精度 3D 打印文件，全过程有友好的 UI 交互。**

### 架构总览

│  ├─ 参数滑块 UI（实时调整 → 重新生成）                              │
│  ├─ 知识库管理（模板 CRUD、示例浏览、标准查询）                       │
│  └─ 可打印性报告 + 打印配置                                        │
├─────────────────────────────────────────────────────────────────┤
│  Backend API (FastAPI)                                           │
│  ├─ /api/generate — 生成管道入口（SSE 流式进度）                    │
│  ├─ /api/templates — 参数化模板 CRUD                              │
│  ├─ /api/standards — 工程标准查询                                  │
│  ├─ /api/export — 格式转换（STEP/STL/3MF）                        │

│  Core Engine                                                     │

### 核心设计原则：流程自由可配置

**过多步骤叠加不一定效果更好。** V3 的每个增强步骤（Best-of-N、多视角、OCR、RAG、可打印性检查等）均有独立开关，用户可根据零件复杂度和场景自由组合。

| 原则 | 说明 |
|------|------|
| **每步可选** | 所有增强步骤独立开关，无强制依赖 |
| **合理默认** | 三个预设：⚡快速（最少步骤）/ ⚖️均衡（推荐）/ 🎯精确（全部增强） |
| **透明代价** | 每个选项配 Tooltip（`?` 图标 hover）：做什么 / 适合什么 / 额外耗时成本 |
| **渐进披露** | 基础用户看预设按钮；高级用户展开自定义面板 |

这确保用户能找到**当前场景下的最优质量/速度/成本平衡**，而非被迫使用"全家桶"。

### 精度保障机制

V3 的核心设计是**让 LLM 做意图理解，让参数化引擎做精确计算，人在环路做参数确认**：

```
V2 精度 = VL准确率 × LLM代码正确率 = 90% × 70% ≈ 63%
V3 精度 = 用户确认率 × 模板正确率 = ~100% × ~99% ≈ 99%
```

### 与 V2 的关系

**渐进式重构，非推倒重来。** V2 约 75% 代码可复用：

| V2 模块 | V3 演进 |
|---------|---------|
| pipeline.py（管道骨架） | 保留，增加新 Stage |
| agents.py（代码执行器） | 直接复用 |
| validators（验证体系） | 保留 + 增加体积/拓扑/可打印性 |
| DrawingSpec 数据模型 | 扩展为 PreciseSpec |
| ModelingStrategist | 升级为 TemplateSelector |
| SmartRefiner | 保留（图片输入场景） |
| 知识库 TaggedExample | 逐步替换为 ParametricTemplate |

## 分阶段路线

按 **质量提升收益 / 实施难度** 排序，分 6 个 Phase：

### Phase 1：架构重构 + 快速收益（基础层）

**目标：** 前后端分离架构 + 立竿见影的质量提升

| 工作项 | ROI | 难度 | 说明 |
|--------|-----|------|------|
| 前后端分离架构搭建 | 基础 | 中 | FastAPI + React + Three.js |
| STL/3MF 导出 | 高 | 低 | CadQuery 已有能力，仅需封装 |
| 体积估算验证 | 中 | 低 | 5-10 行代码 |
| 3D 预览 Viewer | 高 | 中 | 后端 STEP→glTF + Three.js 渲染 |
| Token 用量监控 | 数据 | 低 | LangChain callback |
| 评测基准框架 | 前提 | 中 | 量化后续所有优化效果 |

**精度提升：** 体积估算可过滤 ~15% 的结构性错误
**交付物：** 可独立运行的前后端分离应用 + STL 下载 + 3D 预览

### Phase 2：生成质量引擎（精度攻坚）

**目标：** 管道级质量提升，精度从 3-5% → 1-2%

| 工作项 | ROI | 难度 | 说明 |
|--------|-----|------|------|
| Best-of-N 多路生成 | 高 | 低-中 | N=3 时正确率 40%→78% |
| 多视角渲染 | 高 | 低-中 | 消除单角度盲区 |
| 回滚机制 | 中 | 低 | 防止越改越烂 |
| 拓扑验证 | 中高 | 中 | 检测孔数/阶梯数结构性错误 |
| 结构化 VL 反馈 | 中 | 低 | JSON issues 列表 → 精准修复 |
| 截面分析 | 中 | 中 | 精准验证旋转体轮廓 |

**精度提升：** Best-of-N + 多视角 + 拓扑验证综合提升 ~40%
**交付物：** 增强的生成管道 + benchmark 对比报告

### Phase 3：参数化模板 + 知识库管理（消除结构性错误）

**目标：** 模板覆盖的零件精度 < 1%，知识库可视化管理

| 工作项 | ROI | 难度 | 说明 |
|--------|-----|------|------|
| features 结构化模型 | 高 | 中 | 消除最大技术债（dict → Pydantic） |
| ParametricTemplate 引擎 | 高 | 中高 | 模板 + 参数注入，消除结构性代码错误 |
| 7 种 PartType 模板（~20 个） | 高 | 中 | 每种类型 2-3 个模板变体 |
| 知识库管理 UI | 高 | 中 | 模板 CRUD、示例浏览、在线验证 |
| 向量检索替代 Jaccard | 中 | 中 | embedding + pgvector 语义匹配 |
| 知识库扩充（每类型 5-10 示例） | 高 | 中 | 代码质量地基 |

**精度提升：** 模板覆盖的零件类型精度 < 1%
**交付物：** 参数化模板引擎 + 知识库管理界面

### Phase 4：交互式工作流 + 意图理解（工业级精度）

**目标：** 支持自然语言输入，通过交互达到 < 0.5% 精度

| 工作项 | ROI | 难度 | 说明 |
|--------|-----|------|------|
| IntentParser | 极高 | 高 | NL → IntentSpec → PreciseSpec |
| 工程标准知识库 | 高 | 高 | 螺栓/法兰/公差/键槽/齿轮标准 |
| 参数确认 UI（表单 + 滑块） | 极高 | 高 | 实时参数调整 + 3D 预览刷新 |
| 约束检查引擎 | 中高 | 中 | 参数间工程一致性验证 |
| 可打印性检查 | 高 | 中 | 壁厚/悬挑/孔径/特征尺寸 |
| 打印配置管理 UI | 中 | 中 | FDM/SLA/SLS 配置文件 |

**精度提升：** 用户确认参数 → 精度 < 0.5%
**交付物：** 对话式输入 → 参数确认 → 精确建模的完整工作流

### Phase 5：AI 增强 + 多模态（精度天花板突破）

**目标：** RAG + 多模态输入 + 高级分析

| 工作项 | ROI | 难度 | 说明 |
|--------|-----|------|------|
| RAG 检索（170K 数据集） | 高 | 中高 | Few-shot 提升 LLM 代码质量 |
| OCR 辅助标注提取 | 高 | 中 | 解决标注漏读（图纸输入最大失败源） |
| 多模型投票 | 中高 | 低 | 2-3 个 VL 模型 + 多数投票 |
| 参考图片理解 | 中高 | 高 | 复用 V2 VL + 交互确认 |
| 成本优化（模型降级 + 缓存） | 中 | 中 | 20-30% 成本节省 |
| 两阶段分析 | 中 | 中 | 全局结构 + 局部标注分开读取 |

**精度提升：** RAG 提升 LLM 代码质量 ~30%，OCR 消除标注漏读
**交付物：** RAG 增强管道 + 多模态输入支持

### Phase 6：高级能力（长线投资）

**目标：** 领域深度能力 + 前沿 AI 技术

| 工作项 | ROI | 难度 | 说明 |
|--------|-----|------|------|
| 微调管道（SFT + GRPO） | 高 | 极高 | 参考 NeurIPS 2025 方法 |
| 渐开线齿轮参数化 | 中 | 中 | 齿轮从展示用升级为功能性 |
| 复杂零件模板（sweep/loft） | 中 | 中高 | 扩展零件复杂度上限 |
| 高级可打印性（支撑策略/方向优化） | 中 | 高 | 材料用量 + 打印时间预估 |
| 多材料支持 | 低-中 | 高 | 3MF 多材料定义 |
| 轮廓叠加比对 | 中高 | 中 | 线框图 vs 图纸叠加 |

**精度提升：** 微调模型 Median CD 降低 48%+
**交付物：** 专业级 CAD 生成能力

## 技术选型

| 决策点 | 选择 | 理由 |
|--------|------|------|
| CAD 内核 | **CadQuery 2.x / OCCT** | 工业级精度 + ~450K LLM 训练数据，无可替代 |
| 后端框架 | **FastAPI** | async-first，SSE 原生支持，与现有技术栈一致 |
| 前端框架 | **React 18 + TypeScript** | 与主项目一致，Three.js 生态最好 |
| 3D 渲染 | **Three.js / React Three Fiber** | 后端 STEP→glTF 转换，前端加载 glTF 渲染（见 ADR-4） |
| 向量数据库 | **pgvector** | 不引入新基础设施，PostgreSQL 扩展 |
| LLM 编排 | **LangChain** | 现有使用，callback 机制便于监控 |
| LLM 提供商 | **多模型可配置** | qwen-vl-max(VL), qwen-coder(代码), GPT/Claude(fallback) |
| 参数化引擎 | **自研模板（Jinja2 + CadQuery）** | 灵活性最高，与知识库深度集成 |

## 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| CadQuery WASM 不成熟 | 浏览器端 3D 预览受限 | 使用 Three.js 加载转换后的 glTF/STL，STEP 解析留在后端 |
| 参数化模板覆盖不足 | 新零件类型仍需 LLM 自由生成 | 保留 CodeGenerator 作为 fallback，模板 + 自由生成双轨 |
| 170K 数据集质量参差 | RAG 检索到低质量代码 | 预过滤 + 执行验证 + 人工审核 top-K |
| 微调需 GPU 资源 | Phase 6 启动成本高 | 先用 RAG 验证方向，确认 ROI 后再投入训练资源 |
| 工程标准库覆盖度 | 初期仅覆盖常用标准 | 优先覆盖 80/20 标准（螺栓、法兰），渐进扩展 |
